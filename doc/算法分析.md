# 算法分析、均衡指标设计与场景推荐

## Context

当前实验场景为 50 jobs / 2 server farms / 6 servers，评估 5 种启发式算法与 IDQN/VDN/QMIX/MAPPO/MADDPG。
用户希望：
1. 明确各算法优劣势与预期表现
2. 引入能同时体现"能耗 + 任务完成"平衡的指标
3. 推荐能突出算法差异的额外场景设置

**实现范围**：分析 + 在 `exp1_compare_baselines.py` 和 `exp_compare_all.py` 中增加均衡指标（ESS）。

## Part 1：算法分析（50 jobs, 2 farms, 6 servers）

### 启发式算法

| 算法 | 核心逻辑 | 优势 | 劣势 | 预期表现 |
|---|---|---|---|---|
| **Random** | 随机选 farm/server | 无偏基准 | 无任何优化 | 各指标最差 |
| **Round-Robin** | 轮询 farm/server | Jain's J ≈ 1.0（完美公平） | 完全忽略服务器效率 | 公平性最高，HER 低，energy 中等 |
| **Least-Loaded** | 选利用率最低的 | 减少拒绝（负载均摊） | 不考虑 efficiency_tier，会激活低效旧服务器 | 低拒绝率，energy 中等偏高（分散激活过多服务器） |
| **Best-Fit** | 选最小剩余容量 | 高资源利用率，可减少 active servers | 可能集中在少量服务器导致过载 | ASR 中等，能耗比 Least-Loaded 略好 |
| **Energy-Greedy** | 优先效率等级高的服务器（贪心） | 能耗最优（单步贪心），HER 高 | 短视：高效服务器被填满后拒绝率显著上升 | **energy 最优的启发式，但拒绝率易上升** |

**核心矛盾**：Energy-Greedy 与 Least-Loaded 之间存在天然权衡，前者节能后者降拒绝。没有启发式能同时两者最优。

### MARL 算法

| 算法 | 协作机制 | 优势 | 劣势 | 预期表现 |
|---|---|---|---|---|
| **IDQN** | 无（独立 DQN） | 训练简单 | 两智能体学习过程互不透明（非平稳性），难以协调 farm 选择与 server 选择 | 优于 Random，协作效果不如 CTDE 方法 |
| **VDN** | 线性 Q 分解 Q_tot = ΣQ_i | 有合作信号 | 线性假设限制表达力；服务器功耗模型含二次项，线性分解难以捕捉 | 优于 IDQN，但受限于线性分解 |
| **QMIX** | 单调混合网络（非线性），全局状态输入 | 最强 Q-based CTDE，可捕捉 farm-server 非线性协作；功耗模型的非线性与其混合网络匹配 | 单调约束可能遗漏某些最优策略 | **Q-based 方法中表现最好** |
| **MAPPO** | 集中式 Critic + PPO 更新 | 无价值分解约束；on-policy 适应性强；稳定 | 样本效率低于 off-policy 方法 | **整体最强或与 QMIX 持平**，尤其擅长非贪心的复杂策略 |
| **MADDPG** | off-policy Actor-Critic，集中式 Critic | 样本复用高效 | **为连续动作空间设计**，离散调度动作需要适配（Gumbel-softmax 等），可能不稳定 | 潜力与 QMIX/MAPPO 相近，但训练稳定性较弱 |

**关键洞察**：
- CTDE 方法（QMIX/MAPPO/MADDPG）能通过全局状态看到所有服务器效率，从而学会"将 CPU 密集型任务路由到高效服务器"——这是启发式方法做不到的。
- IDQN 的两个智能体（farm agent + server agent）依次决策，非平稳性相对可控，但仍缺乏协调。
- 在 50 jobs 小规模下，QMIX/MAPPO 应能完成有效探索并收敛至稳定策略。

## Part 2：现有指标的盲点与均衡指标设计

### 现有指标缺陷

| 指标 | 问题 |
|---|---|
| `total_price` | 拒绝全部任务 → price=0 → 虚假"最优" |
| `reject_rate` | 全部接受到低效服务器 → rate=0 → 虚假"最优" |
| `EET = price / completed` | 拒绝 40/50 任务，10 任务用极少能耗 → EET 极低但吞吐极差 |
| `HER` | 只衡量路由质量，不反映完成率 |

### 提出：Eco-Scheduling Score（ESS）

**定义**：

```
ESS = ASR × (1 − price_norm)
```

其中：
- `ASR = completed_jobs / total_jobs`（= `scheduling_success_rate`，已有）
- `price_norm = total_price_mean / max_price_mean`（在所有参评算法中归一化）

**性质**：
- 范围 (0, 1)，越高越好
- 拒绝任务 → ASR 降低 → ESS 降低
- 浪费能耗 → price_norm 升高 → ESS 降低
- 两个维度**缺一不可**，无法用单方向优化规避另一方

**对比 EET**：

```
算法 A：完成 40 jobs，price=100 → EET=2.5，ASR=0.8，price_norm=1.0 → ESS=0.0（price最高）
算法 B：完成 25 jobs，price=40  → EET=1.6，ASR=0.5，price_norm=0.4 → ESS=0.30
算法 C：完成 50 jobs，price=90  → EET=1.8，ASR=1.0，price_norm=0.9 → ESS=0.10
```

→ EET 认为 B 最优，但 B 只完成了一半任务；ESS 正确反映了 A 和 C 的完整性更好。

## Part 3：推荐额外实验场景

均为 `num_jobs` 小（≤30）以降低训练成本，场景聚焦性强。

| 场景编号 | 配置 | 突出效果 | 预期胜者 | 对比点 |
|---|---|---|---|---|
| **S1 资源受限** | jobs=30, farms=2, servers=4 | 容量紧张时的拒绝管理 | QMIX/MAPPO | Energy-Greedy 贪心路由→高效服务器过载→拒绝率上升 |
| **S2 极端异构** | jobs=20, farms=2, servers=6（5 old + 1 new） | 发现并利用唯一高效服务器 | CTDE MARL | IDQN/Round-Robin 不优先路由到新服务器 |
| **S3 同质服务器** | jobs=20, farms=2, servers=6（全同一代） | 无异构优势时的纯负载均衡 | Round-Robin ≈ MARL | 验证 MARL 的收益是否来自异构感知 |
| **S4 多 Farm** | jobs=20, farms=4, servers=3 | 层次决策质量 | QMIX/MAPPO（全局视野） | IDQN farm/server 独立决策，协调失效 |

## Part 4：实现计划

### 目标文件

- [exp1_compare_baselines.py](exp1_compare_baselines.py) — 添加 ESS 后处理计算
- [exp_compare_all.py](exp_compare_all.py) — 添加 ESS 条形图可视化

### 具体修改

#### 修改 1：`exp1_compare_baselines.py` — ESS 后处理

在 `_run_main()` 中 `all_results` 聚合完成后、写 CSV 之前，新增：

```python
# Post-hoc ESS computation（需所有算法都跑完才能归一化）
if all_results:
    max_price = max(r["total_price_mean"] for r in all_results)
    for r in all_results:
        price_norm = r["total_price_mean"] / max_price if max_price > 0 else 0.0
        r["eco_scheduling_score"] = round(
            r["scheduling_success_rate_mean"] * (1.0 - price_norm), 4
        )
```

CSV `fieldnames` 动态生成（已有 `list(all_results[0].keys())`），无需修改 fieldnames 代码。

#### 修改 2：`exp_compare_all.py` — 可视化

在 Exp1 图表区域新增一个 ESS 条形图调用（higher is better，颠倒标准方向）：

```python
# 在已有图表的末尾，plot_exp1(df) 函数内追加
if "eco_scheduling_score" in df.columns:
    _bar_chart(
        df_sorted, "eco_scheduling_score", None,
        "Eco-Scheduling Score (Higher = Better Balance)",
        "ESS",
        "exp1_ess.png",
        lower_is_better=False,
    )
```

`_bar_chart` 的 `std_col` 参数需支持 `None`（当前已有 `[0]*len` fallback，需确认）。
