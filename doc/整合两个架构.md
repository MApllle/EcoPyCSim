# Plan: `exp1_compare_baseline_compat_version.py` — 跨版本兼容评测

## Context

新版 `env/cloud_scheduling.py` 在 obs 中加入了 `cpu_slack` 键（per-server 可用 CPU slack），导致 obs 维度发生变化。但 MARL 算法的 `.load()` 会把 `dim_info` 硬编码进网络权重（见 `MAPPO.py:97-100`、`MADDPG.py:50-63`、`IDQN.py:56-58` 等），因此用新维度去加载旧 checkpoint 会在前向推断时形状不匹配。

当前仓库状态：
- `env/cloud_scheduling_old.py` 保留了改动前的观测空间（无 `cpu_slack`）。
- 除了 MADDPG（新 env 训练）之外，IDQN/QMIX/VDN/MAPPO 的已训练模型都对应旧 env 的 obs 维度。
- 现有 `exp1_compare_baselines.py` 只 import 了新 env，混跑时对非 MADDPG 的模型会 crash。

目标：新增 `exp1_compare_baseline_compat_version.py`（不动原文件），按算法类型切换所用 env，使同一张对比表里同时跑通旧 env 模型与新 env 模型。

## Approach

核心思路：**每个 strategy 跑自己的 env**（而不是全部跑同一个 env）。虽然两个 env 的 obs 维度不同，但都继承同一个 `ParallelEnv`，action 空间、info 字段、奖励接口都一致，所以输出指标可以放进同一张表做对比。

路由规则：
- `strategy == "maddpg"` → `env.cloud_scheduling.CloudSchedulingEnv`（新）
- 其他全部（baselines 和 idqn/qmix/vdn/mappo）→ `env.cloud_scheduling_old.CloudSchedulingEnv`（旧）

实现方式：对原文件做结构化拷贝改造，最小改动，不引入新抽象。

## Key Design Decisions

- **Baselines 跑在旧 env**：baselines 无 checkpoint 可以任选，但跟 IDQN/QMIX/VDN/MAPPO 保持同一 env 的结果更能放在一行对比；MADDPG 单独一行。结果表里加 `env_variant` 列提醒读者两组来自不同 env。
- **不做 obs padding/truncation**：既无必要（展平顺序严格依赖 `sorted(keys)`，插入 `cpu_slack` 会把 `efficiency_tiers` 和 `task_cpu` 的偏移全部改掉），也容易踩坑，路由分流是最干净的方案。
- **不改动 `cloud_env_wrapper.py`**：wrapper 用于训练脚本，eval 里各算法直接 `select_action(obs_dict)` / `collect(obs_dict)`，不经过 wrapper。
- **不改动原 `exp1_compare_baselines.py`**：符合用户要求"exp1_compare_baselines.py 的测试不需要考虑 cloud_scheduling.py 的更改"。
