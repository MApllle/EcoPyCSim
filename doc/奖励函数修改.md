3. _get_reward()（397-448），四分量、每项 ∈ [-1,1]

task_outcome：成功 = +1；首发拒绝 = -clip(1 + 0.25·(lost-1), 1, 2)（按实际损失数次线性加重，封顶 -2）；级联自动拒绝 = 0（不再重复惩罚）。
energy_reward = -tanh(Δp)：只惩罚本次放置带来的功率增量，与其他 task 的到达/离开解耦。平滑、无裁剪死区。
hetero_reward：把 efficiency_tier 中心化到 {-1, 0, +1}，老机器放 CPU 密集任务变负反馈，不是原来的"只奖不罚"。
fit_reward = cpu_utilization_rate：鼓励 consolidation（紧凑打包能减少活跃 server 数，间接压低 static power）。
两 agent 的权重按职责差异化：server_farm 偏 energy（跨 farm 能耗分布），server 偏 fit（VM 级放置质量）。
奖励理论范围 ≈ [-0.8, +0.7]，比原来的 [-2, +1.8] 窄但方差更小、credit 更干净。老版本和注释块一并删除；_last_hetero_reward 仍被 info 暴露，字段保持可用。

# 改动2

改动摘要
1. server.py 新增 first_idle_vm_cpu_slack
精确镜像 check_cpu_mem_constraint 的 CPU 判定：找第一个空闲 VM，返回 max(0, 1 - vm.cpu)；没有空闲 VM 返回 0（让 agent 一眼看到"这个 server 用不了"）。

2. server_farm.py 新增 cpu_slack property
向量化所有 server 的 slack。

3. cloud_scheduling.py 两处

observation_space 对 server_farm 加 Box(num_farms, num_servers)，对 server 加 Box(num_servers,)，值域 [0, 1]。
_get_observation 相应填充 cpu_slack 字段。
4. cloud_env_wrapper.py 让 MAPPO wrapper 也把 slack 喂进去。

所有 MARL 算法 (IDQN/QMIX/VDN/MADDPG/MAPPO) 的 flatten_obs 都是 sorted(keys) 拼接，新 key 会自动按字母序滑入，不需要改各算法代码。

验证（示例）：初始 slack=0.9（基础 VM cpu=0.1，意味着 agent 现在能接受的最大 task.cpu = 0.9）；一步成功调度后，该 server 的 cpus_utilization=0.02，但 cpu_slack=0.9 保持不变 —— 因为 slack 看"下一个空闲 VM"，刚占用的那个 VM 已被跳过，这正是我们想让 agent 学的信号。

注意：obs 维度变了，每个 agent 增加 num_servers_per_farm * num_farms（server_farm）或 num_servers_per_farm（server）个数。所有旧 checkpoint 不兼容新 obs，需要重训。下一步建议把 reward 的 fit_reward 替换成 slack-aware 形式（比如放置后剩余 slack 归一化），或者在策略网络上直接做 invalid action masking（cpu_slack[i] < task_cpu 的动作直接屏蔽），配合新 obs 效果会更明显。
