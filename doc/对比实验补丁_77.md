不足
这套代码原来的主要问题不在“没有想法”，而在“实验链路没闭合”。

exp1_compare_baselines.py 和 exp_heterogeneity_ablation.py 之前把评估环境写死成 100/5/50，但训练脚本实际用的是 300/30/210，导致 MARL 模型加载时很容易出现维度不匹配，结果就是主实验名义上比较多算法，实际上常常只跑了传统基线。其次，这两个脚本对 Jain's Fairness 和 ASR 的记录方式有偏差，直接取 episode 最后一帧的 info，而终态通常所有服务器都空闲，所以会虚假地产生 Jain=1.0、ASR=0.0 这种“好看但没意义”的结果。

实验完整性上也有缺口。exp1_compare_baselines.py 最初只稳定覆盖传统 baseline，加上可选 IDQN/MAPPO，并没有真正把 VDN/QMIX/MADDPG 全接进主对比；exp3_scalability.py 也没有把你想要的 ASR/Jain 一起纳入规模实验输出。可视化方面，exp_compare_all.py 只认 rewards.npy，但仓库里很多算法实际存的是 reward.txt，所以学习曲线图并不总能画出来。还有一个小但实际会卡人的问题是 schedulers/marl/maddpg/MADDPG.py 的 logger 初始化过于脆弱，文件占用时会影响模型加载。

这次修改了什么
这次我主要做的是把“指标设计”真正接成“可复现实验”。

我先统一了评估尺度：把 exp1_compare_baselines.py 和 exp_heterogeneity_ablation.py 的环境配置改到和训练脚本一致的 300 jobs / 30 farms / 210 servers，并补了统一的 dim_info 构造逻辑。这样现有的 QMIX 和 MADDPG 权重已经可以正常加载，避免了之前最核心的维度错位问题。

然后我修正了指标统计口径。现在 Jain's Fairness 和 Active Server Ratio 不再取 episode 结束瞬间，而是在整个调度过程中逐步采样，最后输出过程均值；HER 仍然保留 episode 累计结果。这个改动主要落在 exp1_compare_baselines.py、exp_heterogeneity_ablation.py 和 exp3_scalability.py。

同时我把算法接入范围补全了。exp1 现在增加了 VDN/QMIX/MADDPG 的评估入口，exp2 也能对 QMIX/MADDPG 做异构消融评估；如果后续把 IDQN/VDN/MAPPO 模型训练好放到 results/ 下，这套脚本会自动纳入。可视化方面，我给 exp_compare_all.py 增加了对 reward.txt/rewards.txt 的兼容，让学习曲线不再依赖 .npy 单一格式。最后还顺手修了 schedulers/marl/maddpg/MADDPG.py 的 logger 初始化，降低日志文件占用导致评估失败的概率。

一句话概括
你这次的修改不是简单“加几个指标”，而是把实验从“设计上成立”推进到了“代码上能跑、结果上能解释”的状态。现在最大的剩余工作已经不是改代码，而是重新正式跑一遍 exp1 / exp2 / exp_compare_all，并补齐还没训练出的其他 MARL 模型。
